国际化问题

本身国际化好像没什么大的问题，简单来说就是将原本的硬编码转换成通过key到对应的语言文件或数据库中取对应语言的数据即可。

大致的思路就是由用户向后端传递自己所在的地区（语言），后端使用拦截器等组件拦截设置语言，后续的国际化就是根据这个语言选择对应的语言文件或数据库读取对应的数据返回给前端的。

Java中SpringMVC就是这种方式，默认使用的是请求头中的Accept-Language判定语言。

开发上来说难度不大，但是有以下问题是需要注意的：

  1. 对于某些语言如英语，它的单词是具有单复数之分的（具体规则自行百度吧）数量不同那么单词的形式也会有多变化，所以如果有需要填充数量的情况（比如：用户查询有多少条私信，这里的私信数量是需要到数据库查询之后再填充到语言模板的），那么需要注意单词单复数形式的不同（实际上我在做国际化需求的时候这种情况还是比较少的）。

    还有其他的情况，比如英语中的序数词，1st、2nd、3rd、4th······，There is/are 句型的就近原则等等

  2. 语言模板字符串对于变量应该是具名的，也就是我能够指定参数进行填充的，不是一定得遵循变量的顺序。有这种要求是因为不同的语言在语序上不一定相同的。

    比如"你有{}条私信，收到{}个点赞，收到{}枚硬币"，这种方式就无法指定参数进行填充，传递的参数必须按照正确的顺序才可以返回正确的数据，当不同的语言语序不同的时候，数据会填充错误的。而"你有{0}条私信，收到{1}个点赞，收到{2}枚硬币"这种方式就是可以根据指定参数进行填充，代码中的参数位置固定不变，在不同的语言文件模板中调换下参数的位置即可实现语序的不同的需求，或者像"你有{messageCount}条私信，收到{likeCount}个点赞，收到{coinCount}枚硬币"提供具名参数的形式，参数不存在有序性且可读性会更好。

  3. 做国际化当然就会牵扯到时区的问题，那么数据库这些地方就不能用Date字符串来存时间，必须使用时间戳来存储（其实也不是不行，只是我们必须指定一个时区作为标准，实际上时间戳也是以0时区作为的标准而已）

    不同国家的用户使用的时候根据用户所在的时区或者用户自己设置的地区的时区来转换成对应的时间。

这类问题我觉得只能进行冗余存储了，根据条件判断给出对应的单词形式

Java

Java中似乎并没有相关的解决方案，Java中对于国际化的支持似乎也只有Spring了。（目前我是没有找到其他关于国际化的解决方案，已有的国际化方案也没有考虑这种情况）

2022.5.4补充

MessageFormat提供了很强的模板字符串功能，里面的ChoiceFormat（选择表达式）就可以满足需要了。

不过这个表达式语法不够灵活，所以还是自己实现一套表达式的语法，参考go-i18n的方式（留个坑，有时间自己实现下）

Golang

go-i18n这个国际化框架有考虑这种情况。

指定参数填充

Java

Java的MessageFormat就可以满足需求，根据参数的索引取对应的参数进行填充

或者可以使用Hutool的StrUtil.format，它有个重载方法可以使用Map作为参数，根据key来填充参数
